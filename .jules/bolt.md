
## 2024-05-24 - Optimization: Caching transparency function per layer for FOV compute
**Learning:** The visibility system computes FOV repeatedly using a nested function to fetch tiles and check their transparency. Profiling shows that function setup and grid index validation during raycasting is a hot path (approx 30% of FOV compute time). Python index out-of-bounds error handling is slightly faster than pre-validating coords.
**Action:** Used a dictionary in `VisibilitySystem` to cache the transparency function per `layer_index` per frame, and switched to using a `try...except IndexError` structure for bounding checks inside `is_transparent`. Applied the same optimization to `AISystem` which also calculates line-of-sight. The result is a ~20% speedup in transparency checks.
## 2024-05-24 - Optimization: Caching bounds and arrays for FOV compute
**Learning:** A previous learning suggested `try...except IndexError` was faster for FOV out-of-bounds checks. However, explicit bounds checking is significantly faster (over 6x faster in out-of-bounds cases and ~10-20% faster in in-bounds cases) *if* the max bounds (`len(layer_tiles)`) and the reference to the nested array (`layer.tiles`) are pre-calculated outside the hot-loop closure. The overhead of repeatedly raising `IndexError` exceptions inside the hot loop (FOV raycasting generates many OOB checks at the edges) and repeatedly calling `len()` was the real bottleneck.
**Action:** Extract array references (`layer.tiles`) and calculate bounds (`max_x`, `max_y`) outside of nested functions/closures (like `is_transparent`) that are called repeatedly in hot loops. Use explicit bounds checking (`0 <= y < max_y`) instead of relying on `try...except`.
