
## 2024-05-24 - Optimization: Caching transparency function per layer for FOV compute
**Learning:** The visibility system computes FOV repeatedly using a nested function to fetch tiles and check their transparency. Profiling shows that function setup and grid index validation during raycasting is a hot path (approx 30% of FOV compute time). Python index out-of-bounds error handling is slightly faster than pre-validating coords.
**Action:** Used a dictionary in `VisibilitySystem` to cache the transparency function per `layer_index` per frame, and switched to using a `try...except IndexError` structure for bounding checks inside `is_transparent`. Applied the same optimization to `AISystem` which also calculates line-of-sight. The result is a ~20% speedup in transparency checks.

## 2024-06-25 - Optimization: Re-evaluating Bounds Checking Overhead in Hot Loops
**Learning:** While testing optimizations to `_make_transparency_func`, it was found that standard `try...except IndexError` inside an inner loop called millions of times is significantly slower than explicit bounds checking when exceptions actually happen. Profiling showed that explicitly pre-calculating list bounds (`width`, `height`) and using `if 0 <= y < height and 0 <= x < width:` is over twice as fast as the exception-based approach when handling frequent out-of-bounds queries, which is common in FOV raycasting. Negative indexing in Python is also a dangerous trap when replacing explicit bound checks with `try...except`.
**Action:** Reverted the advice to use `try...except IndexError` for bounding checks in hot loops. Applied explicit pre-calculated length boundary checks to `game/systems/visibility_system.py` and `game/systems/ai_system.py`. Ensure that `try...except` is kept outside of the innermost loops where exception frequency is low.
