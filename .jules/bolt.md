
## 2024-05-24 - Optimization: Caching transparency function per layer for FOV compute
**Learning:** The visibility system computes FOV repeatedly using a nested function to fetch tiles and check their transparency. Profiling shows that function setup and grid index validation during raycasting is a hot path (approx 30% of FOV compute time). Python index out-of-bounds error handling is slightly faster than pre-validating coords.
**Action:** Used a dictionary in `VisibilitySystem` to cache the transparency function per `layer_index` per frame, and switched to using a `try...except IndexError` structure for bounding checks inside `is_transparent`. Applied the same optimization to `AISystem` which also calculates line-of-sight. The result is a ~20% speedup in transparency checks.

## 2024-05-25 - Optimization: Reverted Exception-based bound checking to Precalculated Bounds
**Learning:** While `try...except IndexError` was previously documented as faster than pre-validating coords in the transparency closure for FOV, benchmarking revealed that *pre-calculating* grid bounds (`width` and `height`) outside the closure and using explicit integer comparison (`0 <= y < height`) inside the closure is actually ~2-3x faster than relying on exceptions. The Python exception overhead is substantial when hit frequently, which happens a lot on the edges of the grid during FOV casts.
**Action:** Updated `VisibilitySystem.get_is_transparent` and `AISystem._make_transparency_func` to cache the target layer slice, `height`, and `width` locally before defining the closure. Will avoid exception-based control flow in hot raycasting loops in the future.
